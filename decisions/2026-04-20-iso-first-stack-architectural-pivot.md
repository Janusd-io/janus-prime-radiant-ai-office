# Decision: ISO-First Stack Architectural Pivot

**Date:** 20 April 2026  
**Owner:** Michael Bruck  
**Status:** Resolved  
**Confidence:** High  

## What

Bonaventure reordered the CEO Executive Management System project. The primary deliverable is no longer the weekly CEO meeting and executive dashboard. Instead, it is the **ISO-driven workflow and automation foundation across all twelve functions**, with the management system as a natural byproduct.

## Options considered

1. **Original path (v2):** OKR-driven, top-down design. CEO meeting → strategic initiatives → quarterly objectives → weekly status. Executive dashboard is the primary outcome.

2. **ISO-first path (chosen):** Task-based, bottom-up design. Simon defines ISO standards → departments document workflows → AI identifies automation gaps → agents orchestrate cross-workflows → real-time continuous compliance. Executive visibility is the byproduct.

## Why this decision

Bonaventure's reframe landed for four reasons:

1. **Task-based framing resonates more than OKR hierarchy.** Bonaventure: *"I'm more on task right now on the project level. Seeing where everyone is right now, that's really it."* He already knows where everyone is via direct conversation; what he wants is organisational visibility of *what's been done*, not a CEO dashboard.

2. **ISO compliance is now the gating constraint.** Simon is in motion on ISO 9001 + 27001 + 42001. Every automation must sit on documented, signed-off workflows. This must come before any tool or system design.

3. **Real-time compliance is a strategic commercial play.** Bonaventure's explicit vision: build a continuous ISO compliance engine that Janus can eventually sell back to the industry as the "Janus standard." This requires automation that sits on solid process foundations, not a CEO UI.

4. **Standardisation unlocks automation.** Bonaventure: *"You can't automate what isn't standardised — Simon's work unblocks the AI Office's work."*

## Impact on design

- **Phase 1 is no longer "stand up the CEO Monday meeting."** Phase 1 is "work with Simon to define ISO standards, then per-department workflow documentation with agent assist, then automation identification, then rollup visibility."
- **Every department is Layer 1 simultaneously.** Not top-down rollout from CEO. Parallel workflows per department, with CEO aggregation layer emerging naturally.
- **Tooling decision becomes precondition.** v3 of the design doc needs clear recommendation: Notion alone, Monday.com alone, or two-tool (Monday.com ops SoR + Notion documentation layer).
- **Simon is critical path for ~1 week.** His ISO scope, standards selection, and workflow documentation tool preference unlock everything downstream.

2. Notion vs Monday.com tooling decision locked in.
3. Bonaventure's answers on questions #2, #4, #5 from v2 (meeting frequency, between-meeting visibility, error tolerance).
