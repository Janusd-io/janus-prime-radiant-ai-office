### в‘Ў Inputs

| Category | Examples |
|---|---|
| **Requirements** | Specs gathered from meetings or Slack threads вҖ” must be complete before build starts |
| **Constraints** | Existing system / stack / infrastructure / budget |
| **Access** | Credentials, third-party API keys, sandbox environments |
| **Direction** | Strategic priorities and approvals from Michael / leadership |

### в‘ў Activities (real flow вҖ” 6 steps)

| # | Sub-process | Description | Control point on exit |
|---|---|---|---|
| **1** | **Gather requirements** | Meetings or Slack threads until scope is complete | Requirements signed off В· no build starts before this gate |
| **2** | **Build in sandbox** | Develop the feature / system in an isolated environment | Working build deployed to sandbox |
| **3** | **Stress test** | Try to break it вҖ” functionality, UI/UX, security, APIs, stability | All five test areas pass; weaknesses logged |
| **4** | **Fix & enrol** | Patch what testing surfaced; finalise the build | All known issues resolved or accepted |
| **5** | **Document** | Write SOP, README, implementation plan | All three documents complete |
| **6** | **Handover to IT** | IT department reviews and accepts; deploys company-wide | IT acceptance recorded В· production deploy executed |

### в‘Ј Outputs

| Output | Form |
|---|---|
| **Sandbox-tested software** | Working feature / system / automation / AI agent вҖ” verified by stress testing |
| **SOP** | Standard Operating Procedure document for the new system |
| **README** | Setup and usage documentation |
| **Implementation plan** | Step-by-step deployment plan handed to IT |
| **Documented evidence** | Test logs В· security check results В· API behaviour records В· access logs |
| **Decisions** | Tooling choices В· architecture approvals В· vendor selections |

### в‘Ө Receivers of Outputs

| Type | Receiver |
|---|---|
| **Primary** | IT department вҖ” receives the handover package and deploys the system company-wide |
| **Secondary (internal)** | End-users in the requesting department / team |
| **Subsequent processes** | C5 Service Delivery & Operations В· M3 Performance Monitoring & KPI В· M4 Internal Audit В· M5 Management Review & Corrective Action |
| **External (when applicable)** | Clients В· partners В· certification body (audit evidence) |

### в‘Ҙ Controls & check points

| Stage | Control |
|---|---|
| **Before build (after step 1)** | Requirements complete and confirmed вҖ” no build starts without signed-off scope |
| **Build (step 2)** | Done in isolated sandbox вҖ” never directly in production |
| **Stress test (step 3)** | Functionality coverage В· UI/UX walkthrough В· security checks В· API behaviour under load В· attempts to break the system |
| **Pre-handover (after step 5)** | SOP + README + implementation plan must all exist before handover |
| **Handover gate (step 6)** | IT department reviews and accepts before company-wide deploy |
| **AI-specific (42001)** | AI Impact Assessment when the build introduces a new AI system В· entry added to AI Systems Register |

### в‘§ Monitoring & Measurement (current state + asks)

I don't currently track formal KPIs. My quality bar is **"can I break it?"** вҖ” I try to find weaknesses myself before users do. The five test areas:

| Test area | What I check |
|---|---|
| **Functionality** | Does every feature work under expected load? |
| **UI / UX** | Is it usable, accessible, and intuitive? |
| **Security** | Does it survive standard attack patterns and API abuse? |
| **APIs** | Response times, error rates, edge cases under load |
| **Stability** | Does it stay up under sustained use? |

If I can't break it across those five areas, it's ready for handover.

**Open ask to ISO lead:** I'd appreciate help defining formal KPIs for these criteria вҖ” pass/fail thresholds, measurement frequency, target values вҖ” so the test phase produces something the PULS dashboard can monitor automatically. Suggested KPI candidates the dashboard could track once defined:

| Candidate KPI | Possible source |
|---|---|
| Test-area pass rate (5/5 required for handover) | Manual log вҶ’ Linear вҶ’ PULS |
| Critical security findings at handover | Security scan output |
| Time from requirements gathered to IT handover | Linear cycle data |
| Post-handover defect rate (issues found by IT or users) | Linear issue tracker |
| AI Systems Register coverage (% of deployed AI tools with entries) | Notion / register audit |
