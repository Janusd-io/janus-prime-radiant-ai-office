## §C — Process-improvement candidates this lint

**Memory-system-reference convention (new).** The wiki currently has 4-6 references like `[[feedback-attribution-from-fireflies]]` and `[[project-lint-workstream-backlog]]` from `pulse/2026-05-21-lint*.md` and `entities/vendors/fireflies.md`. These point at memory-system files (under `MEMORY.md`'s tree at `.../memory/`), not wiki vault files. They show as broken refs in lints because the memory directory isn't indexed.

Three options:
1. **Strip the wikilink syntax** — use plain text like *"per the memory entry feedback-attribution-from-fireflies"* instead of `[[feedback-attribution-from-fireflies]]`. Loses the linking affordance but matches the broken-ref expectation.
2. **Whitelist the memory-prefix** — `[[memory:feedback-attribution-from-fireflies]]` and have the lint scan recognise the `memory:` prefix. Adds a syntax convention.
3. **Accept as broken-by-design** — document in CLAUDE.md §6 that memory references are an accepted broken-ref class.

Recommend option 3 (lowest-friction; matches existing curator behaviour). Codify in next CLAUDE.md schema bump.

**Lint script regex refinement.** Two false-positive classes surfaced this session:
- Backticked code spans containing `[[wikilinks]]` — handled by stripping backticked spans before scanning. Already added to this scan's regex; preserve in future lints.
- Markdown-table cells with escaped pipes (`[[slug\|alias]]`) and escaped brackets — the lint counted `[[slug\]]` and `[[slug\]` as broken slugs. The escape characters are Obsidian-table convention; the wikilinks resolve correctly. Worth a regex refinement that normalises `\|` and `\]` before slug extraction.

**Sources-vs-Linear-IDs convention.** The 133-broken-`sources:` count was almost entirely Linear ticket IDs (`air-*`, `aio-*`) and personal-vault-import slugs. The convention works; the lint script should either skip these patterns or treat them as separate signal. Codifying in CLAUDE.md §4 (`sources:` field rules) would make this explicit.
